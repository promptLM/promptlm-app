# promptLM · UI handoff

Two self-contained mini-sites packaged as design specs for Claude Code to
implement against the real backend.

```
handoff/
├── webui/        # The end-user web app: prompt catalog, detail, edit, diff, form
├── report/       # The static report generated by `promptlm report` on every push to main
├── playbook/     # Design ↔ schema ↔ API mapping (start here)
├── api-client/   # Imported @promptlm/api-client source for reference
├── shared/       # Tokens shared across webui + report
└── README.md     # this file
```

Each subfolder is a working prototype: open `index.html` in a browser, click around,
all routes work. Data is hardcoded inside the JSX files — that's intentional. The
shape of that data **is the contract**.

## Start with the playbook

`handoff/playbook/index.html` is the single page that ties the design surfaces
to the real schema and API. Open it first — every UI surface has a page that
lists the operations it calls, the `PromptSpec` fields it binds to, and the
validation rules. The schema and API references are lifted from the imported
`@promptlm/api-client` source under `handoff/api-client/`.

## How to run

```
cd handoff/webui   # or handoff/report
python3 -m http.server 8000
open http://localhost:8000
```

A floating route bar at the bottom of the screen lets you navigate between pages.
Remove that route bar before shipping (it's marked `.route-bar` in `index.html`).

## What ships in v1, what doesn't

The designs include affordances for capabilities promptLM doesn't have yet.
**Treat the design as the target, but ship lean — disable, hide, or strip
anything that isn't backed by real data.** Specifically:

| Capability | Status | Recommendation |
|---|---|---|
| Prompt form (create + edit) | 🟡 v2 WIP | two known bugs documented in playbook/surfaces/form.html |
| Prompt catalog (list, filter, search) | ✅ implemented | ship as-is |
| Prompt detail (spec view) | ✅ implemented | ship as-is |
| Revision history | ✅ implemented (git log) | ship as-is |
| Spec-level diff | ✅ implementable from spec parse | ship as-is |
| **Dev execution metrics** (runs, latency p50/p95, tokens) | ⚠️ requires capture during `promptlm run` / CI | see "Execution capture" below |
| **Cost per run / per 1k tokens** | ⚠️ derived from tokens × model price table | trivial once tokens are captured |
| **Evaluations** | ❌ not implemented (Pro / future feature) | strip from this round, leave the section as a clearly-labelled "coming soon" surface or remove entirely — your call |
| **Production observability** | ❌ explicitly out of scope | not in any design |

### Execution capture

The detail page and report assume promptLM captures `executions[]` per spec —
one entry per `promptlm run` / CI invocation. Each entry:

```ts
{
  id: string,            // e.g. "exec_4f9c"
  when: ISO8601,
  rev: string,           // which revision of the spec produced this run
  author: string,        // git committer or whoever ran the CLI
  context: string,       // "CI · pre-merge" | "local · promptlm run" | etc
  fixture: string,       // path to the input file, e.g. "fx/refund-window.json"
  ms: number,            // wall time
  tin: number,           // tokens in (rendered prompt)
  tout: number,          // tokens out
  ok: boolean,
  error?: string,
}
```

This is local-only data — captured on the developer's machine or in CI, stored
alongside the spec, **never sent anywhere**. It's how the report and detail
page give a sense of "is this prompt expensive / slow / flaky" without any
production telemetry.

If execution capture isn't ready for this round, ship with empty `executions[]`
and the metrics card will gracefully degrade (or hide it).

## Data contract

Both the webui and the report can be backed by the same JSON document:

```ts
type Specs = {
  repo:      { url: string, default_branch: string },
  head:      { sha: string, message: string, author: string, when: ISO8601, pr?: number },
  prompts:   PromptEntry[],
  commits:   Commit[],     // git log of prompts/
  authors:   Author[],     // aggregated
}

type PromptEntry = {
  name: string,            // "doc-rag-answer"
  group: string,           // "rag"
  path: string,            // "prompts/rag/doc-rag-answer.toml"
  description: string,
  spec: PromptSpec,        // current HEAD
  history: Revision[],     // newest first
  executions: Execution[], // newest first, capped at e.g. 100 in the UI
}

type Revision = {
  rev: string,             // "r34"
  tag?: string,            // "v1.8.0" if there's a release
  sha: string,
  author: string,
  when: ISO8601,
  msg: string,             // commit message
  kind: 'add' | 'edit' | 'remove' | 'rename',
  spec: PromptSpec,        // snapshot at this rev (needed by the diff tool)
}

// PromptSpec mirrors the existing TypeScript type in
// components/promptlm-web-ui/src/features/prompt-editor/types.ts
```

Keep this shape — both apps' rendering code is structured around it.

## Iteration loop

1. Claude Code implements against this spec, swapping the in-file sample data
   for a real loader (fetch `/specs.json` or read from the running app's API).
2. Visual changes / new screens come back to the design project as updates
   to the JSX files in `handoff/`.
3. Drop-in replace.

## What's intentionally NOT included

- **Build tooling** — these are unbundled JSX + Babel-standalone. Drop the
  components into the existing webui's React build, or split the report
  into a tiny standalone bundle (Vite's HTML mode is fine).
- **CSS isolation** — `tokens.css` defines globals scoped under `:root`.
  When merging into the existing webui, namespace them or merge with the
  existing token system.
- **Real router** — the hash router in each `index.html` is just enough to
  click around. Replace with React Router / TanStack Router etc.
- **Accessibility audit** — focus rings, aria labels, keyboard nav have not
  been audited. Worth a pass before shipping.
- **Empty / loading / error states** — sample data assumes everything is loaded
  and well-formed. Add empty states for: zero prompts, zero history, zero
  executions, missing fixtures, network failure on diff.
