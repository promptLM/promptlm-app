Release modes

Two server modes (direct vs pr_two_phase) are surfaced via the X-PromptLM-Release-State response header:

  • released — toast says "Released v1.9.0"; flow ends.
  • requested — toast says "Release pending PR merge" with link out to the PR. The form re-arms a small "Finalize release" affordance that calls completeReleasePrompt after merge.

Toast contract

  • On success: toast bottom-center, sticky 6s. Body: "Released vX.Y.Z". Action: "View execution" → opens last execution in Test tab Output column.
  • On failure: no toast — error stays in the side rail so the user can retry without losing context.

Notes & PR scope

  • PR 1 ships the button, the rail, the state machine, gate logic, and the toast — all wired to mock save/release that always succeeds. Easy to flip a flag in the JSX to exercise the blocked states.
  • PR 2 wires real updatePromptSpec + releasePrompt + completeReleasePrompt, and routes the toast's "View execution" action into the Test tab using real executions[].
  • Replaces: the old "Save & release" CTA documented elsewhere in the playbook is gone in #95. Treat any remaining mention as legacy.
